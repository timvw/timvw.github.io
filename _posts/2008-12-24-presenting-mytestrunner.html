Here are a couple of reasons why i dislike the Unit Testing Framework</a> that comes with Visual Studio Team System</a>:</p>

  • Not all versions of Visual Studio are capable of running the tests.</li>
  • Test inheritance is not supported.</li>
  • Running tests via mstest.exe is slow.</li>
  • Visual Studio creating tens of .vmsdi files.</li>
    </ul>

    There are already a couple of better frameworks out there, and currently MbUnit</a> is my favorite one, certainly in combination with TestDriven.NET</a>.</p>

    I have created a custom implementation of the Microsoft.VisualStudio.QualityTools.UnitTestingFramework assembly. Actually, the assembly only has a couple of Attributes for the moment but contributions are always welcome ;)</p>

    In order to achieve better performance i decided to implement a custom test runner. Currently Gallio</a> uses mstest.exe but there might be a day that i decide to write a plugin so that mytestrunner can be used instead.</p>

    I was inspired by Creating Self-testing Assemblies</a> and decided to use that approach but for some odd reason visual studio insists on running unit tests although the assembly is a console application :( Anyway, i can still invoke mytestrunner via the external tools as following:</p>

    The source code is available at http://code.google.com/p/mytestrunner/</a>.</p>
